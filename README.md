# NeuroAuto MVP
Документация проекта в бизнес-языке

## 1. Что это за проект
`NeuroAuto` — это MVP AI-чат-бота для автодилера, который:
- принимает обращения клиентов на сайте;
- отвечает на частые вопросы по базе знаний;
- определяет намерение клиента (кредит, трейд-ин, тест-драйв, сервис, HR);
- автоматически передает обращение в Bitrix24 по нужному маршруту.

Проект создан, чтобы сократить потери на первом контакте, ускорить реакцию на заявки и повысить конверсию в воронке.

---

## 2. Бизнес-цель и результат
### Цель
Создать набор нейроассистентов для автоматизации ключевых функций компании по продаже автомобилей и улучшения качества коммуникации с клиентами и кандидатами.

### Какие задачи закрывает
- Упрощает обработку входящих запросов клиентов и кандидатов.
- Автоматизирует коммуникацию между отделами.
- Повышает скорость ответа, конверсию и удержание клиентов.

### Ожидаемый эффект
- Меньше потерянных обращений.
- Быстрее передача запроса в профильный отдел.
- Снижение ручной нагрузки на менеджеров первой линии.

## 4. Архитектура простыми словами
Система состоит из 4 блоков:

1. **Сайт и чат-виджет**  
   Клиент пишет вопрос прямо на сайте.

2. **Backend (FastAPI)**  
   Принимает сообщение, определяет намерение, обращается к базе знаний и формирует ответ.

3. **LLM + база знаний (GigaChat + ChromaDB)**  
   Бот отвечает не "из головы", а на основе подготовленного FAQ-контента по отделам.

4. **Bitrix24**  
   Если требуется передача менеджеру — лид фиксируется в CRM и направляется в нужный маршрут.

---

## 5. Как работает процесс (сквозной сценарий)
1. Клиент пишет в чат.
2. Система определяет тему запроса и подбирает релевантный контекст из FAQ.
3. GigaChat формирует ответ в заданном стиле.
4. Если нужна передача менеджеру, бот запрашивает подтверждение.
5. После подтверждения создается/обновляется лид в Bitrix24, назначается маршрут и задача.
6. Менеджер получает обращение с контекстом, клиент — подтверждение в чате.

## 3) Как работает система (end-to-end)

1. Пользователь пишет сообщение в чат на сайте.
2. Frontend отправляет запрос в `POST /api/chat`.
3. Backend:
   - определяет триггер маршрутизации (если есть),
   - проверяет hard-эскалацию,
   - запускает RAG (выбор отдела и retrieval из соответствующей коллекции),
   - отправляет запрос в GigaChat:
     - **зачем:** чтобы бот ответил клиенту обычным понятным языком;
     - **на чем строится ответ:** только на найденных данных из FAQ (RAG) и правилах бота;
     - **что приходит обратно:** готовый текст ответа, который backend сразу отдает на сайт;
     - **если токен закончился:** backend сам обновляет токен и повторяет запрос;
     - **если ключ GigaChat (GIGACHAT_AUTH_KEY) не настроен:** бот пишет, что передаст запрос менеджеру.
4. Если пользователь подтвердил передачу менеджеру:
   - создается лид в Bitrix,
   - пишется handoff в timeline,
   - создается activity,
   - создается задача маршрута (`tasks.task.add`),
   - обновляются route-поля,
   - меняется статус лида на маршрутный (`ROUTE_*`).
5. Пользователь получает ответ в чате.

## 4) Функциональность backend (подробно)

### `POST /api/chat`
Принимает:
- `session_id`
- `message`
- `client_name` (опционально)
- `phone` (опционально)
- `preferred_contact` (опционально)
- `car_model` (опционально)
- `budget` (опционально)

Возвращает:
- `answer`
- `escalated` (bool)
- `trigger_code` (если распознан)
- `lead_id` (если создан лид)

### Триггеры маршрутизации
Маппинг из кода:
- `ROUTING_FINANCE -> finance / finance-dept`
- `ROUTING_TRADE_IN -> trade_in / trade-in-dept`
- `ROUTING_TEST_DRIVE -> sales / test-drive-coordinator`
- `ROUTING_SERVICE_QUESTION -> service / service-dept`
- `ROUTING_SERVICE_DIAGNOSTICS -> service / service-diagnostics`
- `ROUTING_BODY_REPAIR -> service / body-repair-dept`
- `ROUTING_PARTS_SALE -> service / parts-dept`
- `ROUTING_SALES_UPSELL -> sales / sales-upsell`
- `ROUTING_HR_MED -> hr / hr-medical`
- `ROUTING_HR_DOCS -> hr / hr-onboarding`
- `ROUTING_HR_TRAINING -> hr / hr-training`

### Двухшаговая эскалация
- Триггер сам по себе не всегда создает лид.
- Сначала бот уточняет и предлагает передачу менеджеру.
- После подтверждения (`да`, `давайте`, `передайте менеджеру`, `перезвоните` и т.д.) запускается маршрут в CRM.
- Состояние pending хранится в памяти процесса (TTL 20 минут).

### Важное ограничение
- Pending-состояние хранится в памяти Python-процесса.
- При рестарте сервера pending-сессии сбрасываются.

## 5) RAG и ChromaDB: как устроено

### Источники знаний
- `faq_sales.txt`
- `faq_hr.txt`
- `faq_service.txt`

### Чанкинг
- Документ делится по пустым строкам (`\n\n`).
- В индекс добавляются смысловые блоки:
  - с `Вопрос:`
  - с `Триггер:`
  - заголовки `[FAQ ...]`, `[Триггеры ...]`

### Коллекции
- `autosales_faq_sales`
- `autosales_faq_hr`
- `autosales_faq_service`

### Векторизация
- Текущая реализация использует локальную hash-векторизацию (`_hash_embedding`).
- Это быстрый офлайн-вариант без отдельной embedding-модели.

### Устойчивость
- При ошибке Chroma вида `Nothing found on disk` автоматически:
  1. удаляется проблемная коллекция отдела,
  2. пересоздается,
  3. переиндексируется,
  4. повторяется query.
- Если восстановить нельзя — fallback retrieval по пересечению токенов.

## 6) GigaChat: как используется

1. Backend получает access token через:
   - `GIGACHAT_TOKEN_URL`
   - `Authorization: Basic <GIGACHAT_AUTH_KEY>`
2. Запрос в модель отправляется на:
   - `GIGACHAT_CHAT_URL`
3. В модель передается:
   - system prompt (роль + правила + RAG-контекст),
   - user message.
4. При `401` выполняется автоматическое обновление токена и повтор запроса.

Документация API токена:
- [Получить токен доступа GigaChat API](https://developers.sber.ru/docs/ru/gigachat/api/reference/rest/post-token)

## 9) Как подключить бота к сайту

Есть два варианта.

### Вариант A (уже реализован в проекте)
- Использовать встроенный виджет из `frontend/index.html` + `frontend/app.js`.
- Просто развернуть backend, он же отдаёт сайт и API.

### Вариант B (внешний сайт)
Если сайт отдельный (другой frontend), достаточно отправлять запросы на `POST /api/chat`.

Пример запроса из JS:

```javascript
const res = await fetch("https://xxxx.example.ru/api/chat", {
  method: "POST",
```

## 13) Диагностика и частые проблемы

### `ModuleNotFoundError` при запуске
Запускается не тот Python/venv.

Решение:
```bash
.venv\Scripts\activate
uvicorn backend.app.main:app --host localhost --port 8000 --reload
```

### Плавающие ошибки/нет логов
На порту `8000` может висеть несколько процессов.

Решение:
- проверить `netstat -ano | findstr :8000`
- завершить лишний процесс и перезапустить сервер.

### Ошибка Chroma `Nothing found on disk`
В проекте уже добавлено автовосстановление коллекции.
Достаточно перезапустить сервер и повторить запрос.

### В CRM “пустые” поля маршрута
Поля могли не отображаться в форме карточки.
Нужно добавить их через `Выбрать поле` в layout карточки лида.

## 14) Текущие ограничения

- Pending-подтверждения маршрутизации хранятся в RAM процесса.
- Нет отдельной БД для истории сессий (кроме данных в CRM и Chroma).
- Нет отдельной внутренней админ-панели маршрутов.
- Embedding-модель упрощенная (hash), не SOTA-семантика.

## 15) Что можно улучшить дальше

### Архитектура и надежность
- Перенести pending-состояние из памяти в Redis/PostgreSQL.
- Добавить централизованный логгер и `trace_id` для ошибок.
- Добавить retry/circuit breaker для внешних API.

### RAG и качество ответов
- Перейти на полноценные embedding-модели.
- Добавить переиндексацию по расписанию.
- Ввести кэш retrieval и кэш ответов.

### CRM-процессы
- Автообновление `route_status` в `in_progress/closed` по событиям задач.
- SLA-контроль с автоэскалацией.
- Дашборд маршрутов (`/api/routing/status`).

### Продукт и UX
- Форма контактов в чате перед передачей менеджеру.
- Индикатор online/offline backend на фронте.
- Мультиязычность и A/B-тесты промптов.

### Безопасность
- Ограничить CORS в продакшене.
- Скрыть служебные endpoint за auth.
- Регулярная ротация webhook и секретов.
