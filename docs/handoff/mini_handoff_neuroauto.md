## 1. Project Overview (кратко)

| Аспект | Содержание |
| --- | --- |
| **Цель** | MVP AI-чата для автодилера: ответы по базе знаний (RAG), определение намерения, маршрутизация обращений в Bitrix24 с задачами и полями маршрута. Снизить потери на первом контакте, ускорить реакцию и повысить конверсию в воронке. |
| **Пользователь** | Посетитель сайта с чат-виджетом (потенциальный покупатель/клиент сервиса; отдельно в продуктовой логике — кандидаты по HR через свои FAQ). |
| **Ключевая ценность** | Первый ответ без «воды» на основе FAQ; предсказуемый тон и формат; при триггерах — явная подготовка к передаче менеджеру и фиксация лида в CRM после подтверждения (см. двухшаговая логика в `main.py`). |

**Стек по контуру:** FastAPI, GigaChat, ChromaDB (RAG по отделам), Bitrix24 webhook.

## 2. Черновик Prompt Engineering Guide (1–2 ключевых промпта)

Ниже два связанных уровня: **что реально собирается в рантайме** и **полная спецификация v3.1** для выравнивания с бизнесом и CRM. В Prompt Engineering Guide обычно отдельно фиксируют **параметры вызова модели**, **ограничения и краевые случаи**, **историю экспериментов/A/B** — для GigaChat в коде задано следующее.

**Параметры инференса (код):** в `backend/app/services/gigachat.py` для чата используются `temperature: 0.2`, `max_tokens: 700` — при смене модели или «размытии» ответов править здесь и фиксировать в changelog карточки промпта.

### 2.1. Промпт в продакшене: `data/bot_prompts.json` + сборка в `FaqRagService.build_system_prompt`

**Роль:** системное сообщение для GigaChat склеивается из JSON-полей и динамического блока RAG.

**Параметры и «ручки» (что править в первую очередь):**

| Параметр | Где задаётся | Зачем |
| --- | --- | --- |
| `system_role`, `style_rules`, `hard_rules`, `sales_objectives`, `response_shape`, `fallback_message` | `data/bot_prompts.json` | Тон, жёсткие запреты, цели диалога, формат ответа, текст при нехватке данных |
| Часы дилера, SLA менеджера | `.env` / `settings` (подставляются в текст системного промпта) | Ожидания по времени ответа человека |
| Отдел для RAG | Авто по запросу + опционально `preferred_department` из триггера (`main.py`) | Какая коллекция FAQ/Chroma используется |
| Глубина контекста | `retrieve(..., top_k=5)` в `rag.py` | Сколько чанков FAQ в промпт |
| Температура / лимит токенов | `gigachat.py` (`temperature`, `max_tokens`) | Детерминированность и длина ответа |
| Лимиты API GigaChat | квоты провайдера, учётная запись | Rate limits — в operational-доках и мониторинге ошибок |

**Ограничения (заложенные в правила):**

- Ответы только из выданного FAQ-контекста; без выдуманных цен и условий.
- Не запрашивать паспорт и чувствительные данные в чате.
- Короткий формат (ориентир 2–4 предложения), один уточняющий вопрос или один CTA.

**Когда работает плохо:**

| Симптом | Вероятные причины |
| --- | --- |
| «Пустые» или общие ответы | В RAG не попало ничего релевантного (`Контекст не найден.` / слабые чанки FAQ). |
| Неверный тон или слишком длинно | Дрейф модели; ослаблены `style_rules` / `response_shape`; нет регрессионных проверок на типовых фразах. |
| Противоречия с менеджером | FAQ устарел или размечен неоднозначно; модель «дотягивает» формулировки — усилить `hard_rules` и актуализировать `data/faq_*.txt`. |
| Лишние темы (медосмотр и т.д.) | Уже есть узкое правило в `hard_rules` — при новых утечках добавлять аналогичные запреты. |

**Связь с кодом:** `backend/app/services/rag.py` (`build_system_prompt`), вызов LLM — `backend/app/main.py` (`gigachat.ask`).

### 2.2. Эталон / полный текст: Auto Sales Assistant v3.1 (`prompts/sales/sales_bot_v3.1_CLEAR_card.md`)

**Роль:** целевой **system prompt** уровня продукта: CONTEXT, INPUT, EXPECTATIONS, ROUTING-таблица, LIMITATIONS, OUTPUT, блок интеграции с CRM.

**Параметры (плейсхолдеры):** подставляются из конфигурации дилера; черновик переменных — `prompts/config/sales_variables.yaml` (компания, адрес, каналы, Bitrix-методы маршрутов, лимиты).

**Ограничения:** только факты из БЗ; при триггерах — ответ из ROUTING + передача полей в CRM; без юридических обещаний и без давления в продажах.

**Когда работает плохо (типично на стыке продукта и ops):**

| Симптом | Причина |
| --- | --- |
| Неверный маршрут (финансы vs трейд-ин) | Триггеры в словарях/правилах не покрывают реальную речь клиента; нужен аудит формулировок и регрессия по `routing`. |
| «Обещания» CRM | Плейсхолдеры `{{ROUTING_*_METHOD}}` не совпадают с реальными webhook-операциями в Bitrix. |
| Цены и акции вне БЗ | БЗ не синхронизирована с прайсом; карточка требует фактов только из БЗ — без процесса обновления KB качество нестабильно. |

**Связь с документацией по карточке:** `docs/prompts/sales_bot_card.md` (примеры входа/выхода, тест-кейсы).

## 3. Метрики успеха

### 3.1. Технические

| Метрика | Зачем |
| --- | --- |
| Доступность API (`/api/health`, доля 5xx) | Стабильность сервиса |
| Латентность ответа чата (p50 / p95) | UX и риск таймаутов виджета |
| Ошибки / таймауты GigaChat и Bitrix (rate, по типам) | Зависимость от внешних API |
| Доля ответов с пустым или «Контекст не найден» (если логировать) | Здоровье RAG и FAQ |
| Доля эскалаций и срабатываний триггеров (по коду) | Нагрузка на менеджеров и качество первой линии |

### 3.2. Бизнес-метрики

| Метрика | Зачем |
| --- | --- |
| Конверсия: обращение → квалифицированный лид | Эффект воронки |
| Конверсия: лид → целевой этап в Bitrix | Качество передачи в отделы |
| SLA первого ответа в чате | Скорость реакции бота/процесса |
| Время до передачи менеджеру (при выбранном маршруте) | Операционная эффективность |
| Доля корректно маршрутизированных лидов (выборка/аудит) | Точность маршрутизации |
| Доля диалогов без эскалации (где это целевое поведение) | Автозакрытие простых кейсов |

*(Согласовано с разделом KPI в `README.md`; конкретные целевые числа задаёт бизнес.)*

## 4. Основные риски и ограничения (2–3 + контекст)

1. **Неверная или грубая маршрутизация** — триггеры и ключевые слова не покрывают реальные формулировки; возможны ошибки полей/статусов в Bitrix. *Митигация:* регулярный аудит лидов, расширение словарей, регрессионные сценарии (см. тест-кейсы в карточке промпта и README).

2. **Качество ответов и RAG** — устаревший или неполный FAQ ведёт к отказам «уточню у менеджера» или к галлюцинациям при слабом контроле. *Митигация:* процесс обновления `data/faq_*.txt`, мониторинг пустого контекста, ужесточение `hard_rules` при дрейфе модели.

3. **Ограничения MVP по состоянию и данным** (из `README`): pending-подтверждения маршрута в памяти процесса; история диалогов не в отдельной БД проекта; сбои внешних API (GigaChat/Bitrix). *Митигация:* fallback-сообщения, мониторинг, в дорожной карте — Redis/PostgreSQL для состояний и более сильные эмбеддинги.
